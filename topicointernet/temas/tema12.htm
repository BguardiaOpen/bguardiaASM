JavaScript y VBScript: alcances y limitaciones

Recordaremos que JavaScript es un lenguaje que permite modificar el flujo de salida HTML, de modo que podemos adaptar la información que presenta el navegador. VBScript (del lado del cliente) funciona del mismo modo, solamente que es tecnología específica de Microsoft Internet Explorer.

Las limitantes de este esquema las hemos atacado, hasta ahora, del lado del servidor. Sin embargo, es posible generar un mayor dinamismo explotando nuevas características de los mismos.

Con lo que hemos visto hasta ahora, Java Script o VBScript en el navegador pueden utilizarse para:

  • Generar dinámicamente el código HTML, reduciendo el tráfico entre el cliente y el servidor.
  • Interactuar con el cliente a través de "prompt"
  • Validar formas o reaccionar a eventos dentro de las mismas.
  • Calcular en forma algorítmica el código HTML, como hicimos en los ejemplos de Calendario y Tablas de Multiplicar.

Entre las ventajas de Java Script, podemos mencionar:

  • Se pueden editar e incluir en HTML sin mayor problema, no se tienen que compilar ni requieren herramientas de programación con mayor complejidad.
  • No tenemos dependencia de las capacidades del servidor.
  • Es soportado por Netscape Navigator 2 en adelante y Microsoft Internet Explorer 3 en adelante, lo que actualmente representa arriba del 95% de los navegadores.
  • No afecta la operación o genera errores con navegadores que no lo soporten; simplemente éstos ignorarán el código Java Script.
  • Es seguro, al no tener acceso a los recursos de la computadora excepto los del navegador. Inclusive, no se tiene acceso a datos de páginas externas al sitio de donde se baja el código JavaScript.

VBScript tiene las mismas ventajas, excepto por no ser soportado por Netscape sin ayuda de "plug-ins".

Las desventajas de ambos esquemas incluyen:

  • Es limitado el acceso a los objetos del navegador; no todos los objetos dentro del navegador son modificables.
  • En particular, no se tiene acceso a modificar el HTML (excepto los datos de una forma) después de que se carga la página, sólo durante la carga.
  • Al ser interpretados, pueden resultar lentos e ineficientes. Por ejemplo, un script que funciona bien en Internet Explorer, inclusive usado 15 ó 20 veces, cuando se repite 100 veces en la misma página traba el navegador.
  • Debido a la seguridad que tienen, y su acceso limitado a los recursos de la máquina, a veces resultan insuficientes para hacer todo lo que se desearía.
  • No se permiten diversas acciones sobre sitios alternos, lo que limita aplicaciones muy prácticas basadas en obtener la información de otra página.

Dynamic HTML

Tanto en Internet Explorer 4 y 5 como en Netscape Navigator 4.x, se busca superar las limitantes de JavaScript.

La idea básica del HTML dinámico, en ambos casos, consiste en permitir que el código Java Script modifique el HTML que ya se ha cargado.

Esto se logra de mejor manera en Internet Explorer ya que su modelo de objetos es mucho más completo, cubre casi todas las capacidades del navegador. De cualquier modo, ambos soportan la idea básica, si bien en formas incompatibles (lo que complica la vida de los desarrolladores en Internet).

Entre las aplicaciones encontramos comúnmente:

  • Menúes que se "abren" en forma de árbol de exploración, sin necesidad de conectarse al servidor nuevamente.
  • Interfases ricas tales como pasar opciones de una lista de opciones a otra (clásicamente el seleccionar de una lista cuáles aplican y cuales no a cierta selección).

Como ventaja básica, la página puede modificar su presentación (tanto en formato como en información) en reacción a diversos eventos. Esto facilita la creación de interfases más ricas y prácticas en su uso.

Desafortunadamente, al no ser todavía un estándar de ambos navegadores dominantes, no es fácil aprovechar las ventajas del concepto. Por ello, generalmente los desarrolladores sólo pueden aprovechar HTML dinámico en aplicaciones de Intranet, donde sabemos que la empresa está estandarizada a un navegador y de este modo es posible usar la implementación específica; o bien, haciendo distintas versiones y seleccionándolas mediante la detección del navegador, sea en Java Script o por medio de un componente como el BrowserCapabilities de ASP.

Algunos ejemplos del uso incluyen:

  1. Ayuda contextual.
  2. Menú dinámico
  3. "Gráfica" construida con DHTML

 

Java

Otra de las tecnologías sumamente interesante y poderosa para ampliar las capacidades de interfase con el usuario, es el uso de "applets" de Java.

Java es un lenguaje con mucha base en C y C++, pero totalmente orientado a objetos. A diferencia de C++, todo en Java es un objeto. Esto hace el lenguaje más sencillo, en particular, el uso de apuntadores se abstrae lo que simplifica hasta cierto punto la programación. Sin embargo, no es un lenguaje trivial para programar, es más complejo que Basic.

Otra característica importante de la mayoría de las implementaciones y la definición del estándar de Java, es que no es ni interpretado ni compilado, sino trabaja en un esquema intermedio llamado "máquina virtual". En este esquema, el código Java se compila a un código máquina definido específicamente para adaptarse a Java, pero que no es igual al código máquina del procesador donde se ejecutará; por lo que se requiere un software adicional, llamado "máquina virtual", el cual traduce del código máquina virtual Java al código máquina nativo de la computadora donde se ejecute.

Respecto a los lenguajes compilados a código nativo, esto da como ventaja la portabilidad a nivel binario y no sólo de código fuente, es decir, si creo una aplicación Java en la máquina virtual en una PC con Windows 95, se ejecutará igualmente en Sun, Macintosh, o cualquier otra plataforma donde se soporte la Java VM.

Respecto a los lenguajes interpretados, Java resulta más eficiente, se ejecuta más rápido al no requerir los procesos de análisis sintáctico y gramatical, sino solamente la traducción directa de los códigos máquina Java a máquina nativa.

Como contraparte, requiere una compilación compleja y es más lento que código nativo.

Existen tres grandes rubros de aplicaciones a las que se ha orientado Java:

  • Las aplicaciones Java, donde se puede realizar un sistema como en cualquier otro lenguaje, con las ventajas de un lenguaje orientado a objetos, el acceso a todos los recursos de la máquina, con la más alta portabilidad.
  • Los "applets" Java, donde se ejecuta código Java dentro de un navegador; esto las hace prácticas para la programación en Internet. Se tiene acceso a muchos recursos de la máquina, aún más que los que permite el navegador con JavaScript por ejemplo. Sin embargo, se maneja seguridad con un contexto de "sandbox", es decir, los recursos a los que puede accesar un "applet" están estrictamente restringidos. Por ejemplo, en general no se permite imprimir o escribir en archivos desde applets Java.
  • Los "servlets" Java, donde se ejecuta código Java dentro del servidor Web. Como recordaremos del tema 4, esta es una tecnología alternativa a CGI  y ASP.

En este tema y el siguiente, explícitamente nos enfocamos a las características y uso de Java en forma de applets.

Las ventajas del uso de applets incluyen una interfaz más poderosa con el usuario, inclusive con cierta capacidad de acceso al servidor (solamente a aquel de donde se "bajó" el código del applet)

Entre las desventajas, nuevamente los applets están bastante limitados, por cuestiones de seguridad. Desafortunadamente, a veces la seguridad es exageradamente restrictiva; y por otro lado, las implementaciones actuales no son totalmente seguras, es decir, tanto Explorer como Netscape tienen errores que permiten a los applets ejecutar código indebido, lo que abre la puerta a hackers y programadores de virus, para invadir la seguridad de un sistema cliente o bien generar daños a la máquina del cliente.

Los navegadores principales que soportan Java incluyen a Netscape Navigator desde la versión 2 en adelante, Microsoft Internet Explorer desde la 3 en adelante, HotJava y Opera.

Controles ActiveX

Finalmente, una tecnología propietaria de Microsoft Internet Explorer y de la plataforma Windows en general, pero que nos permite aprovechar al máximo los recursos de la máquina (particularmente la interfase del sistema operativo Windows), es el uso de controles Active X.

Como recordaremos del tema 8, Microsoft definió su estándar de interoperabilidad con objetos que llamamos COM. Entre las distintas formas de utilizar COM, vimos los componentes en el lado del servidor. Los componentes que van del lado del cliente, generalmente con una interfase propia, son llamados "controles ActiveX". De hecho, son los mismos que se utilizan en Visual Basic para construir las interfases.

Entre algunos de los controles más utilizados contamos con:

  • Un calendario, captura de un número o contador con un "spin button", que permite incrementar o decrementar el valor,
  • Conference Control, que permite establecer conferencias mediante Net Meeting.
  • Popup  Menu, que permite por ejemplo la creación de un menú contextual como los que estamos acostumbrados a trabajar con el botón derecho sobre una ventana en Windows.
  • Shockwave y Flash, que permiten el control de contenido multimedia creado con las herramientas de Macromedia.
  • ActiveMovie, que permite incluir videos y audio, con un control de su avance o retraso desde nuestra página.
  • Internet controls...
  • Wallet, que permite a IE implementar la funcionalidad de seleccionar las tarjetas de crédito para un pago en línea, sin tener que recapturar para distintos sitios la información de mis tarjetas.
  • Real Audio, que permite procesar sonido en tiempo real.
  • Image Scan de Wang, que en general está instalado en Windows NT como una aplicación, y puede ser controlado en un cliente para permitir digitalizar una imagen y luego subirla al servidor.

La obvia ventaja de los controles ActiveX, es que contamos con la mayor potencialidad y flexibilidad para hacer casi lo que se desee en el lado del cliente, con la facilidad de que el código se "baja" de nuestro servidor (posiblemente en Internet) lo que evita los problemas de instalar ejecutables en todos los clientes.

La principal desventaja es en el aspecto seguridad: si entramos a un sitio no confiable, y bajamos un control ActiveX, éste podrá hacer lo que quiera en su máquina, por ejemplo: formatear el disco duro, copiar un virus, apagar el equipo, o peor aún, transmitir información delicada hacia donde se desee.

Por ello, se establecen dos alternativas: el uso de ActiveX en una intranet, de modo que haya un control y confianza para ellos;; o bien, el uso de firmas digitales que permiten verificar el autor del control, lo que permite confiar en su uso.

Para Netscape, existe la alternativa de los Plugins, programas externos que se instalan y tienen la mismas ventajas y desventajas de los ActiveX, excepto porque son más difíciles de instalar, no es automático, sino que se obliga al usuario a realizar la instalación (y por consiguiente, también es más difícil que inadvertidamente bajemos código que nos genere un daño).

Desafortunadamente, no hay compatibilidad y nuevamente, tenemos que usar estas características de acuerdo al navegador.
