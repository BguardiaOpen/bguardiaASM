Tercer Proyecto Parcial, Aplicación distribuida

 

Proyecto en equipos de 3 ó 4 personas.

Se entregará el 11 de noviembre del 2003. Se deberá entregar electrónicamente (por correo electrónico, en la cuenta del Campus, no Hotmail por favor) un archivo .ZIP con todos los archivos del proyecto, incluyendo documentación, y en su caso el URL donde está publicado; podrá revisarse en algún momento su ejecución, en máquinas definidas por los alumnos.

Existen dos opciones de proyecto a elegir, un sistema de foros de discusión (Newsgroups) distribuido usando HTTP, o un juego virtual.

Deberá cumplir las siguientes especificaciones:

Foros de discusión

  • El objetivo del  proyecto será proporcionar una interfase Web a los usuarios, para intercambiar ideas en forma de "tópicos" y "mensajes" posteados en los "foros de discusión"
  • Deberá poder accederse desde cualquier navegador conectado, preferentemente sin preferencia de marca aunque se considera aceptable el uso de DHTML para una interfaz más completa.
  • Se contará con una arquitectura distribuida, ya que podrán tenerse varios servidores de Newsgroups, cada uno con su propia base de datos, que intercambien y repliquen mensajes entre sí.
  • Desde el punto de vista de la base de datos de foros distribuidos, será una aplicación de arquitectura Peer to Peer, en la cual cada nodo contendrá algunos newsgroups propios, y otros réplica de otro nodo.
  • Se intercambiarán regularmente los mensajes entre todos los servidores que contengan un newsgroup dado, de tal modo que se unan las aportaciones de todos los servidores en la copia central y de ahí se distribuyan.
  • La información por  tanto está particionada en los distintos servidores, donde el administrador de cada servidor podrá dar de alta y baja los foros deseados.
  • La base de datos contendrá primordialmente dos tablas: la de foros, con un ID del foro (único entre todos los servidores) y su título, así como el nombre o IP del servidor "primario" que se toma como el "dueño" de dicho foro; y la de mensajes, que contendrá los textos "posteados" por los usuarios, con su identificador (único entre servidores), el correo electrónico del usuario que lo envía, el texto, y el ID del mensaje al que contesta si es parte de un "hilo" de discusión.
  • Los foros de discusión pueden o no ser moderados (o ambos), al gusto del desarrollador.
  • La interfase con el usuario será Web (HTML) generado dinámicamente, sea por JSPs o Servlets.
  • La comunicación entre servidores para la replicación puede ser por medio de Sockets a bajo nivel, o HTTP (clase URL o similares, o bien por medio de Web Services)
  • El proyecto estará realizado completamente en lenguajes HTML, Java Script y Java..

Juego virtual

  • El objetivo del  proyecto será permitir que los participantes jueguen en tiempo más o menos real, con una interfase Web, desde computadoras conectadas al servidor central del juego (que puede o no ser uno de los participantes).
  • Algunos posibles juegos (basados en turnos y por ello más simples para el modelo) son:
    • Dominó (2 a 4 jugadores por partida, opción de enfrentamiento directo o por parejas, N partidas simultáneas)
    • Ajedrez (2 jugadores por partida, N partidas simultáneas)
    • Turista (2 a 6 jugadores por partida, N partidas simultáneas)
    • Juegos de conquista, ej. Civilization (2 a N jugadores por partida, N partidas simultáneas)
  • Es posible pero debe cuidarse por su mayor dificultad, realizar juegos no basados en turnos como:
    • Carreras de autos o motos "lado a lado" (2 a 4 jugadores por partida, las partidas simultáneas podrían requerir múltiples servidores)
  • El primer jugador que desee iniciar una partida "creará" la partida, dándole un nombre o ID con el cual se podrán integrar los otros participantes.
  • Un jugador podrá integrarse como "observador" de una partida, sin participar.
  • Será una aplicación de arquitectura Cliente/Servidor, con un servidor maestro del juego.
  • La base de datos contendrá las tablas propias de la información de estado de cualquier juego (por ejemplo en el dominó, las fichas de cada jugador, las fichas tiradas, y los puntos positivos o negativos acumulados), los puntajes (con historial), los participantes y la máquina desde la que se "firman" al sistema; deberá soportar múltiples juegos (es decir, el servidor deberá ser capaz de contener la información central de dos o N juegos que estén jugando simultáneamente los jugadores)
  • La interfase con el usuario será Web generado dinámicamente, sea por JSPs o Servlets; es conveniente el uso de Applets para la interfase final a detalle, considerando que no se debe requerir que sean applets "firmados".
  • La interfase permitirá a los usuarios consultar reportes del desempeño o historia de lo jugado.
  • El proyecto estará realizado completamente en lenguajes HTML, Java Script y Java.

Se considerarán los siguientes aspectos:

  • Interfaz simple y eficiente.
  • Tolerancia a fallas, de tal modo que en caso de perderse la conexión con otra sucursal, se pueda aceptar pagos y posteriormente se realice la aplicación del mismo transmitiéndolo por la red.
  • Uso de JDBC
  • Uso de Servlets o JSP
  • Uso de Sockets para la comunicación entre servidores
  • Factores extra: implementación de la comunicación sobre HTTP; uso de objetos distribuidos.
  • Diseño de la arquitectura distribuida apropiadamente

Deberá funcionar y verse correctamente tanto en Internet Explorer como Netscape versiones recientes.
