Desarrollo de Aplicaciones Avanzadas en Internet

Segundo Proyecto Parcial: Aplicación dinámica del lado del cliente con persistencia basada en el servidor

Proyecto en equipos de 3 ó 4 personas.

Se entregará el 7 de octubre de 2003. Se deberá publicar en algún servidor, por ejemplo "www.brinkster.com"; adicionalmente se deberá entregar electrónicamente (por correo electrónico) un archivo .ZIP con todos los archivos del proyecto y el URL donde está publicado.

Se considerarán los siguientes aspectos:

  • Interfase compleja (imitando capacidades avanzadas de GUIs no soportadas por HTML tradicional), utilizando dibujos acordes a la especificación.
  • Todas las aplicaciones requieren uso de imágenes GIF o JPEG para botones, fichas, cartas, formatos o iconos. No se calificará la calidad de diseño, pero sí se requerirá hacer algo de producción gráfica.
  • Barra de navegación y efectos especiales en los menúes
  • Uso de Style Sheets para generar interfase consistente
  • Incluir al menos 4 páginas HTML independientes y con funciones distintas
  • Uso de tecnologías del lado del cliente, incluso Applets de Java (requerido al menos uno).
  • Uso de tecnología Scripting del lado del servidor (ASP, PHP, JSP o Perl versión scripting, no CGI)
  • Deberá documentarse la estructura del sitio (página por página, la descripción de lo que hace y qué ligas tiene; así como un árbol general de navegación); y de la base de datos (descripción tabla por tabla de su función, así como de sus campos).

Deberá funcionar y verse correctamente en Internet Explorer 5 o superior

Se tomará como base el proyecto del 1er. parcial (según les haya sido sorteado), extendiendo su funcionalidad, particularmente incluyendo bases de datos.

  • Examen de certificación
  • Sistema experto de diagnóstico
  • Organigrama de la empresa
  • Carrito de compras

Las especificaciones detalladas adicionales a las del primer parcial, son:

Examen de certificación

Se extenderá, para soportar varios exámenes, seleccionando al principio qué examen se desea realizar.

Los exámenes y sus preguntas se alimentarán en una base de datos, así como la respuesta correcta. Esto se hará con una interfaz administrativa, protegida con password; y la interfase tomará como base la información de la base de datos.

El resultado final se almacenará en un "histórico", agregando a la calificación, el nombre del usuario que contestó el examen. Podrá contestar el mismo usuario varias veces el examen, y se tendrá un reporte de su historia. Se deberá tener administración de usuarios y roles, indicando qué usuarios pueden administrar el sistema y qué usuarios tienen derecho a contestar el examen.

Sistema experto de diagnóstico

Se agregará la interfaz administrativa para agregar síntomas y diagnósticos, editando en base de datos la máquina de estados que genera el sistema experto.

Deberá tener control de usuarios y roles (administración y consulta); se registrará un histórico de qué usuario capturó qué síntomas (todo el recorrido), y el diagnóstico que se le dio. Al final, el usuario deberá evaluar el diagnóstico, en escala de 1 (muy mal diagnóstico) a 10, con un comentario abierto (no obligatorio) donde explique el porqué de su calificación.

Podrá sacarse un reporte del uso del sistema, útil para mejorar su capacidad de diagnóstico. Este reporte ordenará por calidad del diagnóstico, y permitirá navegar directamente a agregar más preguntas en los estados donde el diagnóstico fue incorrecto, con objeto de hacerlo más preciso.  Finalmente, el reporte mostrará un promedio de la calificación del sistema experto, tanto a nivel global (toda la historia) como dividido por tiempo en días o semanas (el promedio de ayer, hoy, etc.)

Organigrama/directorio de la empresa

Se extenderá para soportar varias empresas; de modo que al inicio, se seleccionará de qué empresa se va a consultar el organigrama.

Se agregarán los catálogos de áreas de la empresa (recursivo) y empleados, administrando así sus datos. De este modo, el organigrama/directorio no será fijo.

Deberá tener control de usuarios y roles (administración y consulta). El usuario de consulta será anónimo. Se registrará un histórico de cuántos "hits" tiene cada usuario (cuántas veces se consultan sus datos); y se podrá consultar un reporte de visitas con este histórico.

Carrito de compras

Se extenderá para soportar varias empresas en un centro comercial, cada una con su propio catálogo de productos y proceso de compra.

Se agregará en base de datos el catálogo administrable de productos y precios; se registrará un histórico de pedidos, donde se almacenarán los productos seleccionados al final del pedido, si se "confirma", así como qué usuario hizo el pedido.  

Deberá tener control de usuarios y roles (administración y hacer pedidos). Se contará con un reporte de todo lo que ha pedido un usuario determinado.
